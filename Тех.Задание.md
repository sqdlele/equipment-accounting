ТЕХНИЧЕСКОЕ ЗАДАНИЕ
на разработку информационной системы комплексного учета и мониторинга жизненного цикла техники подразделения АО «ИРЗ»

1. Общие сведения
•	Наименование системы: «ИРЗ: Мониторинг и учет».
•	Область применения: Подразделения ИТ-поддержки, административно-хозяйственные отделы и склады АО «ИРЗ».
•	Цель разработки: Замена бумажного/Excel учета единой цифровой средой для контроля перемещения, ремонта и эксплуатации оборудования.
________________________________________

2. Назначение и цели создания системы
2.1. Назначение системы
Система предназначена для автоматизации процессов регистрации, распределения по рабочим местам и сервисного обслуживания материальных ценностей (компьютеры, периферия, специализированное оборудование).
2.2. Цели создания
1.	Прослеживаемость: Обеспечение прозрачной истории перемещения каждого актива.
2.	Минимизация потерь: Исключение ситуаций «бесхозного» оборудования.
3.	Оптимизация ремонта: Анализ частоты поломок конкретных моделей техники.
4.	Скорость инвентаризации: Сокращение времени на сверку остатков на складах и в офисах.

3. Требования к функциональности
Система должна быть разделена на функциональные блоки:
3.1. Модуль управления активами (Склад и Офис)
•	Регистрация новых активов с присвоением уникального инвентарного номера.
•	Изменение статуса актива: «На складе», «В эксплуатации», «В ремонте», «Списан».
•	Привязка актива к локации (кабинет, цех) и материально-ответственному лицу (сотруднику).
3.2. Модуль технического обслуживания (Ремонт)
•	Регистрация инцидентов (поломок) с описанием дефекта.
•	Ведение журнала выполненных работ.
•	Автоматический расчет времени нахождения оборудования в нерабочем состоянии.
3.3. Модуль отчетности и аналитики
•	Генерация инвентаризационных ведомостей.
•	Визуализация данных на дашборде (соотношение исправного/сломанного оборудования).
•	Экспорт данных в формат Excel/PDF.
4. Требования к архитектуре и программному стеку
4.1. Стек технологий
•	Backend: Python 3.10+, фреймворк Django.
•	Frontend: JavaScript / Django Templates (в зависимости от требований к интерфейсу).
•	СУБД: SQLite 3 (на этапе внедрения и эксплуатации в рамках подразделения).
•	Слой доступа к данным: Django ORM (для обеспечения независимости кода от конкретной СУБД).
•	Инфраструктура: Docker, Docker Compose.
4.2. Обоснование выбора СУБД SQLite
Для реализации системы в рамках конкретного подразделения АО «ИРЗ» выбрана СУБД SQLite по следующим критериям:
1.	Автономность и портативность: База данных хранится в виде единого файла, что упрощает резервное копирование и перенос системы между серверами предприятия без необходимости администрирования отдельных серверов БД.
2.	Низкие системные требования: SQLite не требует выделенных ресурсов оперативной памяти для фоновых процессов, что позволяет разворачивать систему на имеющихся мощностях ИТ-отдела.
3.	Архитектурная гибкость: Благодаря использованию Django ORM, переход на клиент-серверные СУБД (PostgreSQL, MS SQL Server) возможен путем изменения конфигурационного файла без модификации программной логики.

5. Требования к надежности и хранению данных
5.1. Обеспечение целостности
•	Использование механизмов транзакций Django для предотвращения повреждения данных при записи.
•	Регулярное создание теневых копий файла базы данных средствами планировщика (Cron/Task Scheduler).
5.2. Безопасность
•	Файл базы данных должен располагаться в закрытой директории внутри Docker-контейнера, доступ к которой ограничен правами суперпользователя.
•	Прямой доступ к файлу базы данных по сети должен быть исключен; взаимодействие с данными осуществляется строго через интерфейс приложения.

________________________________________

6. Техническое сопровождение
•	Система должна поддерживать миграции (Django Migrations) для обновления структуры таблиц без потери накопленных данных об активах.
•	В случае масштабирования системы на весь холдинг (АО «ИРЗ»), предусмотрена возможность бесшовного экспорта данных из SQLite в промышленную СУБД PostgreSQL.
