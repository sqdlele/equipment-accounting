# ГЛАВА 2. ПРОЕКТНЫЙ РАЗДЕЛ

## 2.1. Разработка технического задания на информационную систему

Разработка информационной системы «ИРЗ: Мониторинг и учет» начинается с формирования технического задания. Техническое задание определяет назначение системы, область применения, основные функции, требования к данным, надежности, безопасности и условиям развертывания. Наличие технического задания позволяет заранее определить границы проекта и исключить ситуацию, когда программный продукт разрабатывается без четкого понимания ожидаемого результата.

Наименование разрабатываемой системы: «ИРЗ: Мониторинг и учет».

Область применения системы: подразделения ИТ-поддержки, административно-хозяйственные отделы, складские зоны и другие подразделения АО «ИРЗ», связанные с эксплуатацией, выдачей, перемещением, ремонтом и инвентаризацией техники.

Цель создания системы заключается в замене бумажного и табличного учета единой цифровой средой, предназначенной для контроля жизненного цикла техники. Под жизненным циклом в рамках данной работы понимается полный путь оборудования от регистрации в системе до списания: поступление на склад, выдача в эксплуатацию, закрепление за сотрудником, перемещение между локациями, ремонт, возврат в работу или вывод из эксплуатации.

Основными задачами системы являются:

1. Регистрация единиц техники с присвоением и хранением уникального инвентарного номера.
2. Хранение технических и учетных характеристик оборудования.
3. Учет текущего статуса техники: «На складе», «В эксплуатации», «В ремонте», «Списан».
4. Привязка техники к подразделению, помещению и ответственному сотруднику.
5. Ведение истории изменений и журнала перемещений.
6. Регистрация ремонтных заявок и выполненных ремонтных работ.
7. Расчет времени простоя оборудования при ремонте.
8. Формирование отчетов и инвентаризационных ведомостей.
9. Отображение аналитических показателей на дашборде.
10. Разграничение прав доступа пользователей.

Функциональные требования к системе разделены на несколько групп.

Первая группа требований относится к модулю управления активами. Система должна позволять создавать карточку техники, редактировать сведения, изменять статус, выполнять поиск и фильтрацию по инвентарному номеру, названию, категории, статусу и локации. В карточке техники должны храниться инвентарный номер, серийный номер, наименование, модель, категория, производитель, год изготовления, дата приобретения, стоимость, описание, фотография, QR-код, статус, локация и ответственный сотрудник.

Вторая группа требований относится к модулю справочников. Для корректной работы системы должны поддерживаться справочники подразделений, помещений, сотрудников, категорий техники и производителей. Использование справочников позволяет унифицировать данные и уменьшить количество ошибок при вводе.

Третья группа требований относится к мониторингу перемещений. При передаче техники из одного помещения в другое или при смене ответственного сотрудника система должна фиксировать событие в журнале перемещений. В журнале необходимо хранить прежнюю и новую локацию, прежнего и нового ответственного сотрудника, номер документа, комментарий, дату операции и пользователя, оформившего перемещение.

Четвертая группа требований относится к ремонту и техническому обслуживанию. Система должна позволять создавать заявки на ремонт, указывать описание дефекта, тип дефекта, приоритет, исполнителя и статус заявки. Для каждой заявки должна быть возможность добавлять записи о выполненных работах и использованных запчастях. При закрытии заявки система должна фиксировать дату решения и рассчитывать время простоя.

Пятая группа требований относится к отчетности и аналитике. Система должна формировать инвентаризационные ведомости в форматах Excel и PDF, а также отображать на главной странице статистику по состоянию парка техники. К таким показателям относятся общее количество техники, количество активных ремонтов, количество техники на складе, количество техники в эксплуатации, распределение по статусам, категориям и локациям.

К нефункциональным требованиям относятся:

- надежность хранения данных;
- защита данных от несанкционированного доступа;
- работа через веб-браузер без установки отдельного клиентского приложения;
- простота развертывания и переноса системы;
- возможность дальнейшего расширения функциональности;
- поддержка миграций базы данных;
- возможность резервного копирования базы данных и пользовательских файлов.

В качестве серверной части выбран Python и фреймворк Django. Для клиентской части используется JavaScript и статический веб-интерфейс. Для хранения данных применяется SQLite. Для развертывания используются Docker, Docker Compose, Nginx и Gunicorn. Такой набор технологий соответствует задачам дипломного проекта и позволяет быстро развернуть демонстрационную версию приложения.

## 2.2. Проектирование архитектуры системы и структуры базы данных

Проектируемая информационная система реализована как веб-приложение с разделением на клиентскую и серверную части. Такой подход удобен для эксплуатации в подразделении предприятия, поскольку пользователи работают с системой через браузер, а основная логика и данные находятся на сервере. Обновление приложения выполняется централизованно, без необходимости устанавливать новую версию программы на каждый компьютер.

Архитектура системы включает следующие основные компоненты:

1. Клиентская часть, реализованная на JavaScript.
2. Серверная часть на Django и Django REST Framework.
3. База данных SQLite.
4. Веб-сервер Nginx.
5. Контейнерная инфраструктура Docker и Docker Compose.

Клиентская часть отвечает за отображение интерфейса, переходы между страницами, отправку запросов к API и обработку ответов сервера. В приложении реализованы страницы входа, дашборда, списка техники, карточки актива, формы добавления и редактирования техники, журнала перемещений, списка ремонтов, карточки ремонта, отчетов и административного раздела.

[Место для рисунка 2.2 - Скриншот страницы входа в систему]

Рисунок 2.2 - Страница авторизации пользователя

Серверная часть принимает HTTP-запросы от клиента, проверяет права пользователя, выполняет бизнес-логику, обращается к базе данных и возвращает результат в формате JSON или в виде файла отчета. Для построения API используется Django REST Framework. Авторизация пользователей реализована с помощью JWT-токенов: после успешного входа пользователь получает access-токен и refresh-токен, которые используются при последующих запросах к серверу.

Веб-сервер Nginx выполняет роль входной точки приложения. Он раздает статические файлы клиентской части, проксирует запросы `/api/` к backend-приложению, обслуживает статические и медиафайлы. Такая схема приближена к реальному способу развертывания веб-приложений.

Структура backend-приложения разделена на несколько Django-приложений:

- `accounts` - пользователи, роли и права доступа;
- `assets` - учет техники, категории, производители, история изменений и перемещения;
- `locations` - подразделения и локации;
- `employees` - сотрудники предприятия;
- `maintenance` - ремонтные заявки и выполненные работы;
- `reports` - дашборд, Excel- и PDF-отчеты.

Такое разделение повышает сопровождаемость проекта. Каждый модуль отвечает за свою часть предметной области, а взаимодействие между модулями выполняется через модели и API.

Проектирование базы данных выполнено на основе сущностей, необходимых для учета жизненного цикла техники. Центральной сущностью является `Asset` - единица техники. Вокруг нее строятся связи с категориями, производителями, локациями, сотрудниками, историей изменений, перемещениями и ремонтными заявками.

Основные таблицы базы данных:

1. `User` - пользователь системы.
2. `Department` - подразделение предприятия.
3. `Location` - место размещения техники.
4. `Employee` - сотрудник предприятия.
5. `Category` - категория техники.
6. `Manufacturer` - производитель техники.
7. `Asset` - единица техники.
8. `AssetHistory` - история изменений карточки техники.
9. `AssetMovement` - журнал перемещений.
10. `RepairTicket` - заявка на ремонт.
11. `RepairWork` - выполненная ремонтная работа.

[Место для рисунка 2.4 - Схема базы данных с таблицами и связями]

Рисунок 2.4 - Логическая структура базы данных

Таблица `User` хранит данные пользователей системы. Для пользователя указывается роль: администратор, ИТ-инженер, кладовщик или пользователь с правами просмотра. Роль определяет доступные действия. Администратор имеет полный доступ, ИТ-инженер может работать с техникой и ремонтами, кладовщик выполняет операции по учету техники на складе, пользователь с правами просмотра может только просматривать данные.

Таблица `Department` предназначена для хранения подразделений предприятия. Таблица `Location` хранит помещения и зоны размещения техники: склад, офис, кабинет, цех, серверная. Каждая локация может быть связана с подразделением. Это позволяет анализировать распределение оборудования не только по помещениям, но и по организационной структуре предприятия.

Таблица `Employee` хранит сведения о сотрудниках: ФИО, должность, подразделение, email, телефон и признак активности. Сотрудник может быть назначен ответственным за единицу техники или исполнителем ремонтной заявки.

Таблицы `Category` и `Manufacturer` являются справочниками. Категория используется для разделения техники на группы: компьютеры, ноутбуки, мониторы, принтеры, серверы и другие типы оборудования. Производитель позволяет хранить сведения о бренде оборудования и формировать аналитику по моделям и производителям.

Таблица `Asset` содержит основную информацию о технике. Ключевым полем является `inventory_number`, то есть уникальный инвентарный номер. Также хранятся серийный номер, наименование, модель, статус, год изготовления, дата приобретения, стоимость, описание, фотография и QR-код. Через внешние ключи актив связан с категорией, производителем, локацией и ответственным сотрудником.

Статусы актива отражают состояние жизненного цикла:

- `warehouse` - на складе;
- `in_use` - в эксплуатации;
- `repair` - в ремонте;
- `written_off` - списан.

Таблица `AssetHistory` хранит историю изменений карточки техники. При создании актива или изменении важных полей создается запись, где указывается измененное поле, прежнее значение, новое значение, пользователь и дата изменения.

Таблица `AssetMovement` хранит журнал перемещений. В ней фиксируются прежняя и новая локация, прежний и новый ответственный сотрудник, номер документа, комментарий, пользователь, оформивший операцию, и дата записи. Эта таблица необходима для обеспечения прослеживаемости движения оборудования.

Таблица `RepairTicket` хранит ремонтные заявки. Заявка связана с конкретным активом, пользователем, который зарегистрировал проблему, и сотрудником-исполнителем. Также указываются статус, приоритет, описание дефекта, тип дефекта, дата создания и дата решения. В модели предусмотрен расчет времени простоя в часах.

Таблица `RepairWork` хранит записи о выполненных работах по заявке. Для каждой записи указывается заявка, исполнитель, описание работ, использованные запчасти и дата выполнения.

Такая структура базы данных позволяет хранить не только текущее состояние оборудования, но и события жизненного цикла. Это повышает достоверность учета и позволяет формировать отчеты по истории эксплуатации техники.

## 2.3. Алгоритмическое обеспечение функций мониторинга и учета перемещений техники

Алгоритмическое обеспечение системы описывает порядок выполнения основных операций: регистрации техники, изменения карточки, перемещения, создания ремонтной заявки, закрытия ремонта, замены техники со склада и формирования аналитики.

Алгоритм регистрации новой единицы техники:

1. Пользователь авторизуется в системе.
2. Пользователь открывает форму добавления техники.
3. Система загружает справочники категорий, производителей, локаций и сотрудников.
4. Пользователь вводит инвентарный номер, серийный номер, наименование, модель, статус и другие сведения.
5. Пользователь отправляет форму.
6. Backend проверяет корректность данных.
7. В базе данных создается запись `Asset`.
8. Для нового актива автоматически формируется QR-код.
9. В истории изменений создается запись о регистрации техники.
10. Пользователь возвращается к списку техники или открывает карточку созданного актива.

Алгоритм изменения карточки техники:

1. Пользователь открывает карточку актива.
2. Система получает данные актива через API.
3. Пользователь переходит к редактированию.
4. После сохранения backend сравнивает старые и новые значения полей.
5. Если изменились статус, локация, ответственный, наименование или описание, система создает записи в истории изменений.
6. Обновленная информация отображается в карточке актива.

Такой алгоритм позволяет не только изменить текущее состояние техники, но и сохранить следы изменений. Это важно для контроля действий пользователей и восстановления истории эксплуатации оборудования.

Алгоритм учета перемещения техники:

1. Пользователь выбирает единицу техники, которую необходимо переместить.
2. Система определяет текущую локацию и текущего ответственного сотрудника.
3. Пользователь указывает новую локацию и нового ответственного сотрудника.
4. При необходимости пользователь вводит номер документа и комментарий.
5. Backend проверяет, что новая локация или новый ответственный отличаются от текущих значений.
6. В таблице `AssetMovement` создается запись о перемещении.
7. В карточке актива обновляются поля локации и ответственного сотрудника.
8. В таблице `AssetHistory` создаются записи об изменении локации и ответственного.
9. Пользователь видит обновленный журнал перемещений.

Для операции перемещения используется транзакция базы данных. Это означает, что создание записи в журнале и обновление карточки техники выполняются как единая операция. Если на одном из этапов возникает ошибка, изменения не сохраняются частично. Такой подход повышает целостность данных.

Рисунок 2.7 - Журнал перемещений техники

Алгоритм создания ремонтной заявки:

1. Пользователь открывает раздел ремонтов.
2. Пользователь создает новую заявку и выбирает неисправную единицу техники.
3. Указывается описание дефекта, тип дефекта, приоритет и исполнитель.
4. Backend создает запись `RepairTicket`.
5. Если техника не находилась в статусе «В ремонте», ее статус автоматически изменяется на «В ремонте».
6. В истории изменений создается запись о переводе техники в ремонт.
7. Заявка отображается в списке ремонтов и на дашборде как активная.

Автоматическое изменение статуса актива при создании заявки позволяет избежать расхождений между ремонтом и учетом техники. Если техника передана в ремонт, это должно отражаться и в карточке оборудования.

Алгоритм добавления выполненной работы:

1. Пользователь открывает карточку ремонтной заявки.
2. Пользователь вводит описание выполненных работ.
3. При необходимости указывает использованные запчасти.
4. Backend создает запись `RepairWork`.
5. Запись отображается в журнале работ по заявке.

Алгоритм закрытия ремонтной заявки:

1. Пользователь меняет статус заявки на «Решен» или «Закрыт».
2. Backend сохраняет новый статус.
3. Если дата решения еще не указана, система автоматически заполняет ее текущей датой и временем.
4. На основании даты создания и даты решения рассчитывается время простоя.
5. Если актив находится в статусе «В ремонте», система переводит его в статус «На складе».
6. В истории изменений создается запись о смене статуса.

Время простоя рассчитывается по формуле:

T = (D2 - D1) / 3600

где T - время простоя в часах; D1 - дата и время создания ремонтной заявки; D2 - дата и время решения заявки; 3600 - количество секунд в одном часе.

Алгоритм замены техники со склада:

1. Пользователь открывает ремонтную заявку.
2. Система предлагает кандидатов на замену из техники со статусом «На складе».
3. По умолчанию отображаются активы той же категории, что и неисправная техника.
4. Пользователь выбирает заменяющую единицу техники.
5. Backend проверяет, что выбранная техника находится на складе и не совпадает с неисправной.
6. Заменяющая техника переводится в эксплуатацию, получает локацию и ответственного сотрудника неисправной техники.
7. Неисправная техника переводится на склад и освобождается от ответственного сотрудника.
8. В истории изменений фиксируются изменения по обеим единицам техники.
9. В ремонтную заявку добавляется запись о выполненной замене.

Данный алгоритм полезен для практической эксплуатации, так как позволяет быстро выдать сотруднику рабочее оборудование вместо неисправного и сохранить корректный учет обеих единиц техники.

## 2.4. Программная реализация модулей регистрации оборудования и формирования отчетности

Программная реализация системы выполнена с разделением на backend и frontend. Backend отвечает за хранение данных, бизнес-логику, авторизацию, API и формирование отчетов. Frontend отвечает за пользовательский интерфейс, навигацию, формы, таблицы, фильтры и визуализацию статистики.

Backend-приложение реализовано на Django. В файле настроек подключены основные приложения: `accounts`, `assets`, `locations`, `employees`, `maintenance`, `reports`. В качестве механизма авторизации используется JWT. Все основные API-запросы требуют авторизации пользователя. Для работы со списками используется пагинация, фильтрация, поиск и сортировка.

Модуль `accounts` реализует пользовательскую модель. Помимо стандартных полей пользователя, модель содержит роль и телефон. Роли позволяют ограничивать доступ к операциям изменения данных. В системе выделены роли администратора, ИТ-инженера, кладовщика и читателя.

Модуль `assets` реализует учет техники. В нем описаны модели категорий, производителей, активов, истории изменений и перемещений. Для работы с API используются ViewSet-классы. Они позволяют выполнять стандартные операции создания, чтения, обновления и удаления записей. Для списка техники реализованы фильтры по статусу, категории, производителю, локации и ответственному сотруднику, а также поиск по инвентарному номеру, серийному номеру, названию и модели.

При создании новой единицы техники система автоматически создает запись в истории изменений. Также для актива формируется QR-код. QR-код содержит данные вида `IT-ASSET:инвентарный_номер:наименование`. Это позволяет в дальнейшем использовать QR-идентификацию для быстрого перехода к карточке оборудования или сверки техники при инвентаризации.

При редактировании карточки техники backend сохраняет прежние значения важных полей и после обновления сравнивает их с новыми. Если значение изменилось, создается запись в истории. Такой механизм реализован отдельно в сервисной функции, что позволяет использовать его в разных сценариях: при обычном редактировании, перемещении и замене техники со склада.

Модуль `locations` хранит подразделения и локации. Локации имеют тип: склад, офис или кабинет, цех, серверная. Наличие типа локации позволяет точнее описывать место размещения оборудования и формировать понятную структуру предприятия.

Модуль `employees` хранит сведения о сотрудниках. Сотрудник может быть материально ответственным лицом за технику или исполнителем ремонтной заявки. Для сотрудника хранится ФИО, должность, подразделение, email, телефон и признак активности.

Модуль `maintenance` реализует учет ремонтных заявок. При создании заявки система переводит связанную технику в статус «В ремонте». При изменении статуса заявки на «Решен» или «Закрыт» автоматически фиксируется дата решения. На основании даты создания и даты решения рассчитывается простой оборудования. Также модуль поддерживает добавление выполненных работ и замену неисправной техники оборудованием со склада.

Модуль `reports` реализует дашборд и формирование отчетов. API дашборда возвращает следующие данные:

- общее количество техники;
- распределение техники по статусам;
- распределение техники по категориям;
- распределение техники по локациям;
- количество техники на складе;
- количество активных ремонтов;
- последние ремонтные заявки.

На frontend эти данные отображаются в виде карточек и диаграмм. Для построения диаграмм используется библиотека Chart.js. Визуализация позволяет быстро оценить состояние парка техники без ручного просмотра таблиц.

Формирование отчетов реализовано в форматах Excel и PDF. Excel-отчет представляет собой инвентаризационную ведомость с колонками: номер, инвентарный номер, наименование, модель, категория, производитель, статус, локация, ответственный, дата покупки, год изготовления, серийный номер, стоимость и описание. Для удобства восприятия в Excel используются заголовки, цветовое оформление статусов, границы таблицы и ширина колонок.

PDF-отчет формируется как инвентаризационная ведомость ИТ-техники. Он подходит для печати и передачи в неизменяемом виде. Для корректного отображения русскоязычного текста используется поддержка шрифтов.

## 2.5. Организация контейнеризации приложения и развертывание среды

Для удобного развертывания информационной системы используется контейнеризация. Контейнеризация позволяет упаковать приложение, зависимости и настройки окружения в изолированные контейнеры. Это снижает вероятность ошибок при переносе системы между компьютерами и упрощает демонстрацию проекта.

В проекте используется Docker и Docker Compose. Docker отвечает за сборку и запуск отдельных контейнеров, а Docker Compose описывает состав всей среды и связи между сервисами.

Контейнерная среда включает два основных сервиса:

1. `backend` - серверная часть Django-приложения.
2. `nginx` - веб-сервер, обслуживающий frontend и проксирующий запросы к backend.

Сервис `backend` собирается из каталога `backend`. В нем запускается Django-приложение. При старте backend может выполнять миграции базы данных и загрузку демонстрационных данных. Для хранения данных используются Docker volumes: `db_data`, `media_data`, `static_data`. Это позволяет сохранить базу данных, загруженные файлы и статические файлы при перезапуске контейнеров.

Сервис `nginx` собирается на основе конфигурации из каталога `nginx`. Он открывает внешний порт приложения, по умолчанию порт 80. Nginx раздает статические файлы frontend-части, проксирует запросы `/api/` к backend-контейнеру, обслуживает `/admin/`, `/static/` и `/media/`.

Рисунок 2.13 - Контейнерная структура приложения

В `docker-compose.yml` описана внутренняя сеть `app_network`, через которую взаимодействуют контейнеры. Backend не публикуется напрямую наружу, а доступен для Nginx внутри Docker-сети. Пользователь обращается к приложению через Nginx. Такой подход повышает управляемость развертывания и соответствует типовой схеме публикации веб-приложений.

Для настройки приложения используются переменные окружения:

- `RUN_DEMO_SEED` - загрузка демонстрационных данных;
- `DJANGO_SETTINGS_MODULE` - модуль настроек Django;
- `SECRET_KEY` - секретный ключ Django;
- `DEBUG` - режим отладки;
- `ALLOWED_HOSTS` - разрешенные хосты;
- `CORS_ALLOWED_ORIGINS` - разрешенные источники запросов;
- `PORT` - внешний порт Nginx.

Типовой порядок развертывания системы:

1. Перейти в каталог `webapp`.
2. Создать файл `.env` на основе `.env.example`.
3. Указать секретный ключ, разрешенные хосты и CORS-настройки.
4. Выполнить сборку контейнеров командой `docker compose build`.
5. Запустить приложение командой `docker compose up -d`.
6. Открыть веб-интерфейс в браузере.
7. Проверить авторизацию, дашборд, список техники и отчеты.

Для локальной разработки без Docker backend может запускаться через виртуальное окружение Python, а frontend - через простой статический сервер. Однако основным рекомендуемым способом запуска является Docker Compose, так как он обеспечивает одинаковую среду для backend, frontend и Nginx.

Контейнеризация дает следующие преимущества:

- упрощает перенос приложения на другой компьютер или сервер;
- снижает зависимость от локальных настроек операционной системы;
- позволяет быстро развернуть демонстрационную версию для защиты;
- отделяет данные от контейнеров с помощью volumes;
- упрощает обновление приложения;
- позволяет приблизить проект к реальной эксплуатации.

С точки зрения надежности важно, что база данных SQLite хранится в отдельном volume. Это позволяет сохранить данные при перезапуске или пересборке контейнеров. Медиафайлы, включая фотографии техники и QR-коды, также сохраняются отдельно. Статические файлы backend-приложения размещаются в отдельном volume и доступны Nginx только для чтения.
