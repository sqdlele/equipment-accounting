# ГЛАВА 3. ТЕСТИРОВАНИЕ И ЭКСПЛУАТАЦИЯ

## 3.1. Разработка программы и методики тестирования

После завершения основной разработки информационной системы «ИРЗ: Мониторинг и учет» была проведена проверка работоспособности ключевых функций. Цель тестирования заключалась в том, чтобы убедиться, что система выполняет требования, сформулированные в техническом задании: позволяет регистрировать технику, изменять ее статус, вести историю перемещений, создавать ремонтные заявки, рассчитывать простой оборудования, формировать отчеты и отображать аналитические показатели.

Тестирование проводилось с позиции обычного пользователя системы, так как для дипломного проекта важно подтвердить не только наличие программного кода, но и возможность выполнения реальных рабочих сценариев. Проверялись операции, которые могут ежедневно выполнять сотрудники ИТ-поддержки, склада или административного подразделения.

В качестве тестовой среды использовалось веб-приложение, развернутое через Docker Compose. В состав среды входили backend на Django, frontend на JavaScript, веб-сервер Nginx и база данных SQLite. Для проверки использовались демонстрационные данные: пользователи с разными ролями, несколько единиц техники, подразделения, локации, сотрудники и ремонтная заявка.

Программа тестирования включала следующие направления:

1. Проверка авторизации и разграничения доступа.
2. Проверка отображения дашборда.
3. Проверка регистрации новой единицы техники.
4. Проверка поиска и фильтрации техники.
5. Проверка карточки техники и истории изменений.
6. Проверка перемещения техники между локациями.
7. Проверка создания и закрытия ремонтной заявки.
8. Проверка расчета времени простоя.
9. Проверка формирования отчетов в Excel и PDF.
10. Проверка запуска системы через Docker Compose.

Методика тестирования была основана на ручной проверке пользовательских сценариев. Для каждого сценария определялись исходные данные, последовательность действий, ожидаемый результат и фактический результат. Такой подход выбран потому, что система имеет выраженный пользовательский интерфейс, а многие функции важно проверить именно через реальные формы, таблицы, фильтры и кнопки.

Дополнительно для backend-части были подготовлены автоматизированные API-тесты. Они находятся в файле `webapp/backend/apps/assets/tests.py` и проверяют основные сценарии, которые напрямую связаны с темой диплома: создание карточки техники, формирование QR-кода, запись истории изменений, перемещение техники, создание и закрытие ремонтной заявки, получение статистики дашборда и выгрузку Excel-отчета. Эти тесты не являются сложными, но хорошо подходят для подтверждения того, что основные функции системы работают не только через интерфейс, но и на уровне серверной логики.

Для запуска автоматизированных тестов можно использовать команду (в консоли выводится отчёт по каждому из 13 сценариев — его можно использовать как подтверждение для диплома вместе со скриншотами):

`python manage.py test apps.assets`

При запуске через контейнерную среду команда имеет вид:

`docker compose run --rm --entrypoint python backend manage.py test apps.assets`

Ожидаемый результат: новая единица техники сохраняется в базе данных, появляется в общем списке, получает уникальный инвентарный номер и QR-код.

Фактический результат: запись успешно создается и отображается в списке. В карточке техники отображаются введенные сведения. QR-код формируется автоматически при создании записи.

## 3.2. Анализ результатов тестирования и оценка эффективности внедрения системы

По результатам тестирования можно сделать вывод, что основные функции информационной системы «ИРЗ: Мониторинг и учет» работают корректно. Система позволяет выполнять регистрацию техники, хранить карточки оборудования, менять статусы, вести историю изменений, фиксировать перемещения, оформлять ремонтные заявки, добавлять выполненные работы, рассчитывать простой и формировать отчеты.

При этом в ходе проверки были выявлены отдельные технические моменты, которые потребовали доработки. Это нормальная часть разработки, потому что некоторые проблемы становятся заметны только после запуска приложения в полной среде, а не при просмотре отдельных модулей.

Первая проблема была связана с размещением файла базы данных SQLite внутри контейнера. При первоначальном запуске возникала ситуация, когда приложение не могло открыть файл базы данных, если каталог для хранения данных еще не был создан. Для решения проблемы в настройках backend был добавлен механизм создания каталога `data` перед обращением к базе. После этого база данных стала корректно создаваться и сохраняться в Docker volume. Это решение нельзя назвать сложным, но без него запуск в контейнере был нестабильным.

Вторая проблема была связана с отображением русскоязычного текста в PDF-отчете. При использовании стандартных PDF-шрифтов кириллица могла отображаться некорректно. Для решения была добавлена регистрация шрифтов DejaVu, поддерживающих русский текст. После этого PDF-ведомость стала пригодной для просмотра и печати. Этот момент оказался важным, так как дипломная система ориентирована на русскоязычную документацию предприятия.

Третья проблема проявилась при локальной разработке frontend и backend на разных портах. Браузер блокировал отдельные запросы из-за ограничений CORS. Для обхода и нормальной работы были добавлены настройки разрешенных источников запросов. В итоговом варианте основным способом запуска выбран Docker Compose с Nginx, где frontend и API доступны через один адрес, поэтому проблема CORS в рабочем сценарии практически не возникает.

Четвертый момент был связан с логикой статусов после ремонта. Сначала казалось, что после закрытия заявки технику нужно сразу возвращать в статус «В эксплуатации». Однако на практике такая логика не всегда подходит: после ремонта оборудование может вернуться на склад, пройти проверку или ожидать повторной выдачи. Поэтому в системе принято нейтральное решение: после закрытия ремонта техника переводится в статус «На складе». Если ее нужно вернуть конкретному сотруднику, это можно оформить отдельным перемещением или выдачей. Такой вариант выглядит более аккуратным для учета, хотя требует одного дополнительного действия пользователя.

Пятый момент относится к импорту данных из Excel. В реальных таблицах названия колонок могут отличаться, а даты и суммы могут быть записаны в разных форматах. Поэтому в системе была добавлена обработка разных вариантов заголовков и форматов дат. Полностью защититься от плохо заполненных файлов сложно, но текущий вариант позволяет загрузить типовые ведомости и получить список ошибок по проблемным строкам.

В целом тестирование показало, что система готова к демонстрации и может использоваться как основа для внедрения в подразделении. Она не заменяет полностью все внутренние регламенты предприятия, но закрывает основную практическую задачу: централизованный учет техники и контроль ее жизненного цикла.

Эффективность внедрения системы можно оценить по нескольким направлениям.

Первое направление - сокращение времени поиска информации. В ручном учете сотруднику приходится искать сведения в разных таблицах или журналах. В разработанной системе достаточно воспользоваться поиском по инвентарному номеру, названию или модели, после чего открывается карточка техники с актуальным статусом, локацией и ответственным лицом.

Второе направление - повышение прозрачности перемещений. Система хранит не только текущее местоположение оборудования, но и журнал перемещений. Благодаря этому можно восстановить, где техника находилась ранее, кто был ответственным и когда была выполнена операция.

Третье направление - контроль ремонта. Ремонтная заявка связана с конкретной единицей техники. При создании заявки статус оборудования автоматически меняется на «В ремонте», а после закрытия заявки фиксируется время простоя. Это уменьшает риск ситуации, когда техника фактически не работает, но по учетным данным продолжает числиться в эксплуатации.

Четвертое направление - ускорение подготовки отчетности. Инвентаризационная ведомость и журнал ремонтов формируются автоматически. Это снижает объем ручной работы и уменьшает вероятность ошибок при переносе данных в отчетные документы.

Пятое направление - улучшение управленческой информации. Дашборд показывает общее количество техники, активные ремонты, технику на складе и распределение оборудования по статусам, категориям и локациям. Такие данные помогают быстрее оценить состояние парка техники.
